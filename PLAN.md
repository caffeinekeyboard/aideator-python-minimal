# Aideator - LISP to Python Migration Plan

## Context

The legacy LISP code (`legacy/minimal-aideator.lisp`) implements an LLM-powered **idea tree builder**. It uses a tree of "posts" where each node is generated by an LLM based on its ancestors' context. The goal is to migrate this into a modular, fully-functional Python codebase using the Gemini API.

### What the LISP code does

- **Core data structure**: `post` with fields `ptype`, `purpose` (parent), `name`, `description`, `achievers` (children)
- **12 post types**: mission, stakeholder, goal, barrier, cause, solution, abstraction, analogy, inspiration, question, answer, improvement
- **Tree navigation**: `context()` returns ancestor chain, `find_first()` searches ancestors by type
- **LLM integration**: `propose_achiever()` builds type-specific prompts, calls LLM, parses JSON response, creates new posts
- **17 transition rules** governing which child types can be added to which parent types

### Transition Rules

```
mission       -> stakeholder
stakeholder   -> goal
goal          -> barrier, solution, abstraction
barrier       -> cause, solution, abstraction
cause         -> cause, solution, abstraction
abstraction   -> analogy
analogy       -> inspiration
inspiration   -> solution
solution      -> improvement, barrier, question
question      -> answer
```

## Project Structure

```
aideator/

├── tree.py              # context(), find_first(), build_post(), describe_context()
├── transitions.py       # Legal parent->child type rules, validation
├── prompts.py           # All 12 prompt templates (faithful LISP reproduction)
├── llm.py               # Gemini API client (ask + parse_response)
├── engine.py            # IdeaEngine: propose_achiever orchestrator

```

## Module Details

### `models.py`
- `PostType(str, Enum)` with all 12 values including `answer` and `improvement` (which the LISP code uses as distinct ptypes despite omitting from comments)
- `Post(BaseModel)` using Pydantic with UUID-based IDs, self-referential `purpose` parent reference, and `achievers` list

### `tree.py`
Direct translations of LISP tree operations:
- `context(post)` -> ancestor chain `[post, parent, ..., root]`
- `find_first(ptypes, post)` -> first ancestor matching any of the given types
- `build_post(purpose, ptype, name, desc)` -> create and attach new post
- `describe_context(post)` -> human-readable ancestor chain string

### `transitions.py`
- `TRANSITIONS` dict mapping parent type -> list of allowed child types
- `ACTION_NAMES` dict with human-readable names for each transition
- `validate_transition()` and `get_allowed_children()` helpers

### `prompts.py`
All 12 prompt templates faithfully translated from LISP, including:
- Common preamble: "Please propose a new {type}" + context description
- Per-type body with specific instructions and existing sibling deduplication
- JSON response template appended to each prompt
- **Two LISP bugs fixed**: Lines 120 and 293 used `(format nil ...)` instead of `(format stream ...)`, discarding intended text for mission and answer prompts

### `llm.py`
- `LLMClient` wrapping `google-generativeai` SDK with Gemini 2.0 Flash
- Loads API key from `.env` via `python-dotenv`
- `ask(prompt)` sends prompt, returns raw text
- `parse_response(response)` extracts JSON from markdown fences, handles trailing commas

### `engine.py`
- `IdeaEngine` orchestrator combining prompts, LLM, and tree operations
- `create_mission(name, desc)` for root creation
- `propose_achiever(ptype, purpose)` validates transition, builds prompt, calls LLM, parses, and attaches

### `serialization.py`
- `tree_to_dict()` / `dict_to_tree()` for JSON-safe conversion (breaks circular purpose references)
- `export_json()` / `import_json()` for file I/O
- `print_tree()` for numbered text visualization

### `cli.py`
Interactive CLI with commands: `tree`, `select N`, `add TYPE`, `actions`, `info`, `context`, `save FILE`, `load FILE`, `help`, `quit`

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Pydantic over dataclasses | Validation, self-referential model support, JSON ergonomics |
| `answer` + `improvement` as PostTypes | LISP code treats them as distinct ptypes with own prompt templates |
| Fix 2 LISP bugs | `format nil` on lines 120/293 discards intended prompt text |
| UUID-based post IDs | CLI navigation and JSON serialization |
| Gemini 2.0 Flash default | Fast, capable, good at structured JSON output |
| Circular `purpose` reference | Matches LISP semantics; serialization handled separately |

## Usage

```bash
# Install dependencies
pip install -r requirements.txt

# Run the CLI
python main.py

# In the CLI:
# 1. Create a mission (provide name + description)
# 2. Use 'add stakeholder' to add stakeholders
# 3. Select posts with 'select N' and keep building the tree
# 4. Save with 'save output.json'
```
